# Phase 1: Foundation - Research

**Researched:** 2026-03-17
**Domain:** Python monorepo foundation -- shared client library, host resolution, HTTP client, structured output, uv workspace
**Confidence:** HIGH

## Summary

Phase 1 builds the foundation that every future skill depends on: the `claws-common` shared library, the uv workspace configuration, and the project scaffolding. This phase has no external service dependencies (no servers, no ML models) -- it is pure Python packaging and library code.

The core deliverables are: (1) a working uv workspace with root `pyproject.toml`, (2) the `claws-common` package with host resolution, HTTP client, and structured output utilities, and (3) infrastructure files (`.gitignore`, workspace config). Every code pattern established here will be inherited by all future skills and servers.

**Primary recommendation:** Build `claws-common` with three focused modules (`host.py`, `client.py`, `output.py`), use hatchling as build backend, verify pip-installability from git URL in a clean environment early.

### Pattern 1: Root pyproject.toml (Virtual Workspace)

**What:** The root `pyproject.toml` declares a virtual workspace. It is NOT a pip-installable package itself. It manages workspace members and shared dev dependencies.

**Example:**
```toml
[project]
name = "lobster-claws"
version = "0.1.0"
requires-python = ">=3.12"

[tool.uv]
package = false

[tool.uv.workspace]
members = ["common", "skills/*", "servers/*"]

[dependency-groups]
dev = ["ruff>=0.15", "pytest>=9.0", "pytest-httpx>=0.36"]

[tool.ruff]
target-version = "py312"
line-length = 100

[tool.ruff.lint]
select = ["E", "F", "I", "UP"]

[tool.pytest.ini_options]
testpaths = ["common/tests", "skills", "servers"]
```

### Pattern 2: Package pyproject.toml (claws-common)

**What:** Each workspace member has its own `pyproject.toml` with real PyPI dependency names. The `[tool.uv.sources]` section is for local development only and is ignored by pip.

**Example:**
```toml
[project]
name = "claws-common"
version = "0.1.0"
description = "Shared client library for Lobster Claws skills"
requires-python = ">=3.12"
dependencies = ["httpx>=0.28"]

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
```

### Pattern 3: Host Resolution Chain

**What:** Priority-based host resolution: env var override > Docker detection > localhost fallback.

**Example:**
```python
# common/src/claws_common/host.py
import os
from pathlib import Path

def resolve_host() -> str:
    """Resolve the host address for reaching servers."""
    if host := os.environ.get("OPENCLAW_TOOLS_HOST"):
        return host
    if _in_docker():
        return "host.docker.internal"
    return "127.0.0.1"

def _in_docker() -> bool:
    """Detect if running inside a Docker container."""
    if Path("/.dockerenv").exists():
        return True
    if os.environ.get("container") == "docker":
        return True
    try:
        with open("/proc/1/cgroup", "r") as f:
            return "docker" in f.read()
    except (FileNotFoundError, PermissionError):
        return False
```

### Pattern 4: HTTP Client Wrapper

**What:** Thin wrapper around httpx that adds service-aware error messages, configurable timeouts, and structured error handling.

**Example:**
```python
# common/src/claws_common/client.py
import httpx
from claws_common.host import resolve_host

class ClawsClient:
    """HTTP client for skill-to-server communication."""

    def __init__(self, service: str, port: int, timeout: float = 30.0):
        self.service = service
        self.port = port
        host = resolve_host()
        self.base_url = f"http://{host}:{port}"
        self.timeout = timeout

    def get(self, path: str) -> dict:
        """GET request with error handling."""
        url = f"{self.base_url}{path}"
        try:
            resp = httpx.get(url, timeout=self.timeout)
            resp.raise_for_status()
            return resp.json()
        except httpx.ConnectError:
            raise ConnectionError(
                f"Cannot connect to {self.service} server at {url}. "
                f"Is the server running? Check: curl {self.base_url}/health"
            )
        except httpx.TimeoutException:
            raise TimeoutError(
                f"Request to {self.service} timed out after {self.timeout}s ({url})"
            )

    def post_file(self, path: str, file_path: str, **params) -> dict:
        """POST a file with multipart upload."""
        url = f"{self.base_url}{path}"
        try:
            with open(file_path, "rb") as f:
                resp = httpx.post(
                    url,
                    files={"file": f},
                    params=params,
                    timeout=self.timeout,
                )
            resp.raise_for_status()
            return resp.json()
        except httpx.ConnectError:
            raise ConnectionError(
                f"Cannot connect to {self.service} server at {url}. "
                f"Is the server running? Check: curl {self.base_url}/health"
            )
```

### Pattern 5: Structured Output Convention

**What:** Consistent output contract for all CLIs: result to stdout, diagnostics to stderr, exit codes 0 (success), 1 (user error), 2 (infrastructure error).

**Example:**
```python
# common/src/claws_common/output.py
import json
import sys

def result(data: str | dict) -> None:
    """Print result to stdout. Strings print as-is, dicts print as JSON."""
    if isinstance(data, dict):
        print(json.dumps(data), flush=True)
    else:
        print(data, flush=True)

def error(message: str, exit_code: int = 1) -> None:
    """Print error to stderr and exit."""
    print(f"Error: {message}", file=sys.stderr, flush=True)
    sys.exit(exit_code)

def fail(message: str) -> None:
    """User-fixable error (exit code 1)."""
    error(message, exit_code=1)

def crash(message: str) -> None:
    """Infrastructure error (exit code 2)."""
    error(message, exit_code=2)
```

### Anti-Patterns to Avoid

- **Path dependencies in pyproject.toml:** Never use `claws-common = {path = "../common"}`. Use named dependencies + `[tool.uv.sources]` for local dev override. Path deps break pip install from git URLs.
- **Per-service host env vars:** Never create `WHISPER_HOST`, `RESY_HOST`. Use single `OPENCLAW_TOOLS_HOST` for all services.
- **Printing without flush:** Never use bare `print()` in CLI output. Always `flush=True` or use the `output.py` helpers.
- **Using uv-build as build backend:** Container uses pip, not uv. Always use hatchling.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP client | Raw urllib/socket calls | httpx with ClawsClient wrapper | Connection pooling, timeouts, HTTP/2, proper error types |
| Package building | Custom setup.py scripts | hatchling + pyproject.toml | Standards-compliant, works with pip and uv |
| Workspace management | Symlinks, custom scripts | uv workspaces | Single lockfile, dependency resolution, editable installs |
| Python linting | Multiple tools (flake8+isort+black) | ruff | Single fast tool, consistent config |

**Key insight:** Phase 1 is entirely about packaging and library plumbing. Every component has a well-established solution. The risk is not in complexity but in getting the dual-environment (uv dev / pip container) contract right.

## Common Pitfalls

### Pitfall 1: Path Dependencies Break pip Install from Git URLs

**What goes wrong:** `claws-transcribe` depends on `claws-common`. If specified as a local path dep, pip install from git URL fails because paths resolve relative to a temp clone directory.
**Why it happens:** Python path dependencies are non-portable. They work in editable dev installs but fail for any remote install.
**How to avoid:** List `claws-common` as a normal named dependency in `[project.dependencies]`. Use `[tool.uv.sources]` with `workspace = true` for local dev only.
**Warning signs:** Works locally, fails in CI or fresh Docker containers.

### Pitfall 2: stdout Buffering Swallows CLI Output in Docker

**What goes wrong:** CLI prints result but agent sees no output. Python buffers stdout in non-interactive terminals (Docker subprocess invocation).
**Why it happens:** Python's default 8KB stdout buffer. Output less than 8KB that exits without flush is lost.
**How to avoid:** Always use `flush=True` in print calls. Provide `output.py` helper functions that flush automatically. Document `PYTHONUNBUFFERED=1` for container env.
**Warning signs:** CLI works interactively but produces no output when called by agent.

### Pitfall 3: host.docker.internal Resolution Failures

**What goes wrong:** HTTP calls from container get connection timeouts or DNS failures. Error messages are generic networking errors.
**Why it happens:** Multiple failure modes: missing `extra_hosts`, server bound to 127.0.0.1 instead of 0.0.0.0, macOS firewall, server not running.
**How to avoid:** `ClawsClient` catches `httpx.ConnectError` and provides actionable error messages including the service name, URL, and a curl command to diagnose.
**Warning signs:** "Connection refused" errors that vary between machines.

### Pitfall 4: --break-system-packages Dependency Conflicts

**What goes wrong:** Installing claws packages overwrites system Python packages in the node:24-bookworm container.
**Why it happens:** PEP 668 exists to prevent this. `--break-system-packages` bypasses the protection.
**How to avoid:** Keep `claws-common` dependencies minimal (httpx only). Pin versions. Test on fresh node:24-bookworm container.
**Warning signs:** pip warnings about replacing distro-installed packages.

### Pitfall 5: uv.lock Not Committed or Stale

**What goes wrong:** Different developers get different dependency versions. CI fails with version conflicts.
**Why it happens:** `uv.lock` must be committed to git. If regenerated without checking, it may change unexpectedly.
**How to avoid:** Commit `uv.lock`. Run `uv sync` to verify lockfile is current. Add `.python-version` to lock the Python version.
**Warning signs:** `uv sync` produces different results on different machines.

from unittest.mock import patch
from claws_common.host import resolve_host, _in_docker

def test_env_var_override():
    with patch.dict(os.environ, {"OPENCLAW_TOOLS_HOST": "my-host"}):
        assert resolve_host() == "my-host"

def test_docker_detection_dockerenv(tmp_path):
    with patch("claws_common.host.Path") as MockPath:
        MockPath.return_value.exists.return_value = True
        with patch.dict(os.environ, {}, clear=True):
            assert _in_docker() is True

def test_localhost_fallback():
    with patch.dict(os.environ, {}, clear=True):
        with patch("claws_common.host._in_docker", return_value=False):

## Open Questions

1. **pip install order for interdependent packages from git URLs**
   - What we know: `claws-transcribe` depends on `claws-common`. Both install via `pip install git+...#subdirectory=...`. pip needs `claws-common` installed first.
   - What's unclear: Whether pip can resolve this in a single command or requires two sequential commands.
   - Recommendation: Test with two sequential pip install commands. If single-command works, document it; if not, provide an install script.

2. **Python version in node:24-bookworm**
   - What we know: Bookworm ships Python 3.11. The container may have a different version depending on the node:24-bookworm build date.
   - What's unclear: Exact Python version in the current container image.
   - Recommendation: Use `requires-python = ">=3.11"` instead of `>=3.12` to be safe, OR verify the container Python version and adjust. The stack research says 3.12 but this should be validated.
